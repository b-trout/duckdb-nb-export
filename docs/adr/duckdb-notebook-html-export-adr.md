## ADR-001: 実装言語・レンダリングスタックとして Python + Jinja2 + Pygments を採用する

- ステータス: Accepted

### コンテキスト
notebookの読み取り→再実行→HTML描画というnbconvert的パイプラインを実装するにあたり、言語・フレームワークを選ぶ必要がある。候補はPython系(Jinja2/Pygments)とNode.js/TS系(Nunjucks/Shiki)。

### 決定
Python + Jinja2(テンプレート) + Pygments(SQLシンタックスハイライト)を採用する。

### 根拠
- nbconvert自体が同じ構成(Jinja2 + Pygments)であり、設計・実装のノウハウを踏襲しやすい。
- `duckdb` Pythonクライアントが最も枯れており、Arrow/DataFrame連携も容易。
- DuckDB UI本体(TypeScript)との統合を意識するよりも、まず単体で動くものを素早く作ることを優先する。

### 影響
- notebookから直接呼び出す「マジックコマンド」機能(ADR-005)は、Pythonプロセスとの橋渡しが必要になる。
- `duckdb` Pythonパッケージのバージョンがユーザー環境のDuckDBより古いと、storage versionの新しいui.db・対象DBを開けない。依存の上限は固定せず、不整合時は明確なエラーメッセージで更新を促す(design doc 8章)。

## ADR-002: notebookデータへのアクセス方式

- ステータス: Accepted(2026-07-05改訂: WALセットコピーを追加、ロック挙動の検証完了)

### コンテキスト
notebookの内容は `ui.db` 内の `notebook_versions.json` に格納されている。この内部DBはDuckDB UI稼働中はロックされており、実機検証(design doc 6.3#4)により**別プロセスからは read_only ですら開けない**ことが確認された。また非公式スキーマである旨の警告がある。

### 決定
CLI実行時は、`ui.db` と `ui.db.wal`(存在する場合)を**セットでOSファイルコピー**し、コピーを `duckdb` Pythonクライアントで読み取り専用で開いて `notebook_versions` テーブルから対象notebookの `json` カラムを取得する。

- WALをコピーに含める理由: 実機検証(design doc 6.3#5)で、直近の変更がWALにのみ存在し、本体だけのコピーでは最新の編集が見えないことを確認したため。
- 2ファイルコピーはアトミックでないため、コピー後に読み取り検証+リトライ(既定3回)を行う(design doc 4.1節)。
- ロック保持中でも直接開ける前提の代替案は、検証結果により完全に否定された(コピー方式が唯一の現実的選択肢)。

### 検討した代替案
- DuckDB UIが提供するHTTP API経由で取得する: UIサーバーが内部的に叩いているエンドポイントを利用する案。フロントエンドが非公開なためAPI仕様も非公開であり、リバースエンジニアリングコストが高い。ただしADR-005のロック問題の解決策候補(b)として再浮上しており、Phase 1.5の調査対象(design doc 6.2#6)。
- ファイルを直接バイナリパースする: DuckDBのファイルフォーマット自体を自前パースするのは非現実的。DuckDBクライアント経由が妥当。

### 未解決事項
- Windowsでのロック中ファイルコピー可否(Linuxでは可能と検証済み。design doc 6.2#8)。
  - 2026-07-05追記: Phase 1はLinux/macOSのみ正式対応と決定(ADR-008)したため、本検証はPhase 1.5以降に繰り延べる。Phase 1のブロッカーではなくなった。

### 追記(2026-07-05、notebook JSONスキーマ実機調査)
非公式スキーマの実体(`notebooks` / `notebook_versions` / `current_notebook_id` / `has_onboarded` の4テーブルとそのDDL)を実機確認済み(design doc 6.3#9)。本ADRが前提としていた「非公式スキーマである」というリスク自体は変わらないが、具体的なテーブル構造・カラム定義が判明したことで、Reader層の実装・AT-009(前提検証テスト)の期待値が確定できる状態になった。

## ADR-003: セル結果は再実行して取得する(キャッシュされた結果は使わない)

- ステータス: Accepted

### コンテキスト
notebookのJSONにクエリ結果そのものが保存されているかは未確認だが、保存されていたとしても「実行時点のスナップショット」であり、エクスポート時点の最新データと乖離する可能性がある。

### 決定
エクスポート時に各セルのSQLを対象DBに対して**再実行**し、最新の結果を取得してHTML化する(nbconvertの `--execute` に相当する動作をデフォルトにする)。

### 影響・トレードオフ
- 長時間クエリや副作用のあるSQL(DDL/DML)を含むnotebookでは、再実行が高コスト・危険になりうる。安全策(トランザクション戦略・エラー処理・タイムアウト)は**ADR-007**で規定する。
- 将来的に「再実行しない(保存済み結果があればそれを使う)」オプションを追加する余地を残す。

## ADR-004: チャートはクライアントサイドの宣言的埋め込みで再現する

- ステータス: Accepted(方針レベル。具体的ライブラリ選定は未決 — チャート設定JSONの実機調査待ち)

### コンテキスト
DuckDB UIのチャート機能の内部実装(描画ライブラリ、設定JSON構造)は非公開。完全互換の再現は困難。

### 決定
チャートセルについては、取得したクエリ結果と(推測可能な範囲の)チャート設定を、Vega-LiteまたはObservable PlotのようなJS宣言的仕様に変換し、HTML内にインラインで埋め込んでブラウザ側で描画させる。チャート用JSも「単一静的HTML」ゴールを優先してインライン埋め込みを既定とする(design doc 4.3節)。

**Phase 1でのチャートセルの扱い(2026-07-05追記)**: チャート描画に対応するまでの間も、チャートセルのSQLは通常のセルと同様に再実行し、結果を**通常のテーブルとして代替表示**した上で「チャート表示はPhase 1では非対応」と注記する。実行スキップや非表示にはしない — 後続セルがチャートセルのSQL(CTAS等)に依存するケースを壊さないためと、情報の欠落を最小にするため(design doc 4.3節)。

### 検討した代替案
- サーバーサイドで静止画(PNG/SVG)化する(matplotlib等): 生成HTMLが軽量になるが、インタラクティブ性(ホバー、ズーム)を失う。nbconvert HTMLの体験に近づけたいため不採用気味。

### 未解決事項
- チャート設定JSONの構造が完全に不明なため、対応できるチャート種別は限定的になる見込み。最初はテーブルのみ対応し(Phase 1)、チャートはPhase 2で段階的に対応する。
- 2026-07-05追記: 保存形式v3にはチャートセルの表現が存在しないことを実機確認。Phase 1のチャートセル扱い(テーブル代替表示)の対象は、将来フォーマットでチャート表現が追加された場合または別の保存場所が判明した場合に限られる。チャート設定JSONの調査(6.2#2)は保存場所の特定からやり直しになる。

## ADR-005: notebookから「マジックコマンド」的に呼び出す方式

- ステータス: Accepted(2026-07-05改訂: **Phase 1スコープから除外しPhase 1.5へ分離**。二段階アプローチ(案A→案B)自体は維持)

### コンテキスト
単なる外部CLIツールに留めず、DuckDB UIのSQLセルから直接エクスポートを起動できる体験(Jupyterの `%%` マジックコマンド相当)を提供したい。しかしDuckDBの素のSQLには、外部ファイル書き出しや任意コードの実行手段が標準では存在しない。実現には何らかの形で**DuckDB拡張(関数)としてSQLから呼び出し可能な入り口**を作る必要がある。

さらに実機検証(design doc 6.3#4)により、**あるプロセスがRW接続を保持しているDBファイルは、別プロセスから read_only ですら開けない**ことが確認された。サブプロセス方式(案A)は「UIがいま開いているDBに対してセルを再実行する」という典型シナリオで、対象DBに接続できず動作しない。

### 選択肢

**案A: C++/RustのDuckDB拡張から、Pythonレンダラーをサブプロセス起動する**
- 拡張はスカラー関数 `export_notebook_html(name, path)` を公開し、内部でPythonスクリプトをサブプロセス起動する。
- 長所: レンダリングロジック(Jinja2/Pygments)をそのまま流用できる。
- 短所1: 拡張のインストール先環境にPython実行環境と依存パッケージが必要になり、配布が複雑化する。
- 短所2(2026-07-05追記): サブプロセスを起動する拡張はコミュニティ拡張リポジトリの審査を通らない可能性が高く、unsigned拡張として配布することになる。利用者はUIを `allow_unsigned_extensions` 付きで起動する必要があり、「INSTALLしたら動く」体験にはならない。
- 短所3(2026-07-05追記): **対象DBロック問題**。UI稼働中はサブプロセスから対象DBに接続できないため、そのままでは典型シナリオで動作しない。

**案B: レンダリングロジックをC++/Rustで再実装し、単一のDuckDB拡張として完結させる**
- 外部依存を排除し、`INSTALL/LOAD` するだけで動く「正統派」な拡張になる。同一プロセス内で完結するためロック問題も原理的に発生しない。
- 長所: 配布が非常にシンプル。DuckDBのコミュニティ拡張として公開しやすい。
- 短所: テンプレート・シンタックスハイライトをC++で作り直す必要がある。ただしinja/Jinja2CppとDuckDB内部トークナイザの流用により、実装コストは「最も高い」ではなく「中コスト」と見積もる(design doc 4.3節と整合)。

**案C: 「マジックコマンド」を擬似的に実現する(SQLの特殊コメント + 外部ウォッチャー)**
- notebookのセルに `-- @export html` のような特殊コメントを書く運用にし、外部で動くウォッチャープロセスがnotebookの変更を検知してエクスポートを行う。
- 長所: DuckDB拡張の実装が不要、Pythonツールだけで完結する。
- 短所: 「セルを実行した瞬間にエクスポートされる」という同期的な体験にはならず、マジックコマンドと呼ぶには弱い。また対象DBロック問題は案Aと同様に残る。

### 決定
- **Phase 1(MVP)はCLIのみとし、マジックコマンドをスコープから除外する**(design doc 2.1節)。理由は案Aの短所3(対象DBロック問題)が実証され、Phase 1内で解決できる見込みが立たないため。
- マジックコマンドは**Phase 1.5**として分離し、以下の**着手条件**を満たしてから案Aで実装する: UI稼働中の対象DBロック問題の解決策((a)対象DBのスナップショットコピー方式のコスト評価、または(b)UI内部のクエリ実行HTTPエンドポイントへの相乗りの実現性調査)が実機検証で確立していること(design doc 6.2#6)。
- 需要が確認できてから案B(C++移植・コミュニティ拡張化)へ作り直す二段階アプローチは維持する(Phase 2)。

### 未解決事項
- DuckDB拡張(C/C++ API)からサブプロセスを起動することが、拡張のサンドボックス制約上そもそも許可されているか要確認。
- テーブル関数として結果パスを返すのか、単に副作用としてファイルを書くだけで良いのか、SQLインターフェースの形はGoal 2(design doc 2.2節)でスカラー関数+戻り値パスとして確定済み。
- 対象DBロック問題の解決策(a)/(b)の評価(Phase 1.5着手条件)。
  - 2026-07-05追記: **調査は(a)スナップショットコピー方式を優先する**と決定。ui.dbで確立済みのWALセットコピー方式(ADR-002)の延長で技術リスクが低く、非公開仕様に依存しないため。残る評価軸は対象DBのサイズ・整合性・コピーコスト。(b)UI内部HTTPエンドポイントへの相乗りは、(a)が巨大DBでコスト面から不成立と判明した場合のフォールバック候補として残す(design doc 6.2#6)。
- **2026-07-05追記(GitHub Issue #13、解消): notebookコンテキスト伝播は存在しないことを確定**。DuckDB UIフロントエンドバンドル(ui拡張 0dfdf34 / hatchling.11eef943)を静的解析した結果、セル実行は `/ddb/run` へ生SQLテキストをbodyでPOSTするのみで、ヘッダー全集合も `X-DuckDB-UI-Connection-Name`(`connection_<ランダム>` でnotebookとは無関係)・Database-Name・Schema-Name・Parameter系・Result系・Errors-As-JSON・Request-Description・Extension-Versionに限られ、notebookId/cellId系のヘッダーは存在しない。`runCell()` がcellIdを渡す先はクライアント側テレメトリのみで、SQL末尾に付与される `-- MD_SQL_METADATA: {...}` コメントも固定値のみである。したがって `export_notebook_html('current', ...)` の自動解決は実現不可能であり、design doc 2.2節Goal 2に定義済みのフォールバック(明示的なnotebook名/IDを必須とし、ない場合は明確なエラー)を**恒久仕様**に格上げする。なお `_duckdb_ui.main.current_notebook_id` テーブル(design doc 6.3#9)は「最後に開いたnotebook」をタブ切替・新規作成時にのみ更新する永続化であり、セル実行のタイミングとは同期しないため、`'current'` の近似解決に用いることは**採用しない**(複数notebookタブを開いている環境では実行中のセルと異なるnotebookに誤って解決されうるため)。
- **2026-07-05追記(GitHub Issue #14、解消): 対象DBロック問題は(a)スナップショットコピー方式で確定、(b)は打ち切り**。NVMe SSD / duckdb 1.5.4 / Linux上で、RW接続保持プロセスがありWAL未checkpoint変更がある条件下、`shutil.copy2` によるコピー時間を2回の独立実測で測定した(結果は整合)。1GB級はコピー平均0.5〜0.9秒、3GB級は平均3.1〜4.7秒(コールド/ウォームキャッシュで変動)、10GiB外挿で約10〜16秒(ページキャッシュが効かない定常値ベース)。WALのみに存在する未checkpoint変更は、本体+WALセットコピーのread-only openで両サイズ・複数マーカー行にわたり正しく読めることを確認した。制約として、コピー中は対象DBサイズの約2倍の空き容量が一時的に必要であり、実装では事前の空き容量チェック(対象サイズ×2以上、余裕を見て×2.5)ガードと、コピー失敗時のtry/finallyクリーンアップを設計に含める(HDD/NFS/メモリ逼迫環境ではより遅くなりうる)。これによりPhase 1.5のロック問題解決策は**(a)スナップショットコピー方式で確定**し、(b)UI内部HTTPエンドポイント相乗りの調査は打ち切る。**Phase 1.5の着手条件(design doc 2.1節)は達成した**。

## ADR-006: セキュリティ方針(出力パス制御・XSS・シークレット・外部副作用・信頼境界)

- ステータス: Accepted(2026-07-05新設)

### コンテキスト
第1回レビューで出力パス制御・XSS・信頼境界の欠落が指摘され、design doc 5章で方針を定めた。さらに第2回改訂で以下の2点が追加論点となった。

1. **シークレット漏洩**: エクスポートHTMLは共有・配布が前提の成果物であり、セルSQLに `CREATE SECRET` のキー値等が含まれていた場合そのまま漏洩する。
2. **外部副作用**: 実機検証(design doc 6.3#6)で、トランザクション内の `COPY ... TO` はROLLBACKしてもファイルが残ることが確認された。ATTACH先DBへの書き込み・httpfs経由のリモート書き込み・`INSTALL`/`LOAD` も同様に巻き戻らない。「既定では事実上read-only」という旧表現は対象DBファイル内にしか成立しない。

### 決定
1. **出力パス制御**: 出力パスは絶対パスへ正規化(symlink解決含む)した上で、許可ベースディレクトリ外を指す場合は拒否する。文字列ベースの `..` 拒否ではなく正規化後のパス比較で判定する。
2. **XSS対策**: テンプレートエンジンの `autoescape` を既定で有効にし、SQL文字列・結果セル値は常にエスケープして出力する。
3. **シークレットマスキング**: `CREATE SECRET` 文を検出し、**すべてのパラメータ値**(TYPE・PROVIDER・SCOPEのような構造的要素を除く)を一律 `***` にマスクしてHTML出力する(2026-07-05確定: KEY_ID/SECRET/TOKEN等の既知の秘密キー名のみをマスクする案は、新プロバイダ追加時のリスト保守漏れがそのまま漏えいになるため不採用。REGION等の非秘密値も隠れるが実害は小さい)。任意の文字列リテラル内の秘密の網羅的検出は行わない(過剰検出を避ける)。残余リスクは実行前確認プロンプトでの目視を防波堤とし、ユーザー向けドキュメントに明記する。
4. **外部副作用の既定**: 既定では外部アクセスを遮断**しない**。`read_parquet`/`read_csv`/httpfs読み込みはDuckDBの中核ユースケースであり、既定遮断は一般的なnotebookを壊すため。代わりに(a)実行前確認プロンプトでセルSQL一覧を目視させる、(b)`--no-external-access` オプション(`SET enable_external_access = false`)を提供する、(c)ROLLBACKの保護範囲が対象DBファイル内のみであることをドキュメントに明記する。
5. **信頼境界**: 本ツールは呼び出し元の既存権限の範囲でSQLを再実行するのみで権限昇格は発生しない。他者作成notebookのCLIエクスポート時は、実行前にセル一覧を表示し `--yes` なしでは確認プロンプトを挟む。

### 検討した代替案
- 外部アクセスを既定遮断(`--allow-external-access` オプトイン): 安全側だが、外部ファイル読み込みを使う大多数のnotebookが既定で動かなくなるため不採用。
- 書き込み系ステートメント(COPY TO/ATTACH等)のみ検出して追加確認: 体験は良いが検出ロジックの網羅性維持がコストのため不採用(将来の改善候補として残す)。
- シークレットを含むセルのSQL本文を非表示化: 漏洩リスクは最小だが再現性・可読性を損なうため不採用。

## ADR-007: トランザクション・エラー処理戦略(実機検証に基づく改訂)

- ステータス: Accepted(2026-07-05新設。ADR-003の安全策を実装可能な形に具体化)

### コンテキスト
第1回改訂版のdesign docは「既にトランザクション中であればSAVEPOINTを発行」「セル失敗時も後続を継続」を規定していたが、実機検証(duckdb 1.5.4、design doc 6.3節)で以下が判明し、そのままでは実装不可能であることがわかった。

1. **DuckDBはSAVEPOINT/ネストトランザクションをサポートしない**(`ParserException`)。
2. トランザクション内でのエラーは種別により影響が異なる: CatalogException(テーブル不在等)後は継続可能だが、**ConstraintException等の後はトランザクション全体がabort**され、以降の文はすべて `Current transaction is aborted` で失敗する。SAVEPOINTがないため文単位の復旧手段は存在しない。
3. DDL(CREATE/DROP TABLE)はROLLBACKで完全に巻き戻る(安全策の根幹は成立)。
4. `connection.interrupt()` によるクエリ中断は可能で、中断後も接続を再利用できる(タイムアウト実装手段として利用可能)。

### 決定
1. **単一トランザクション方式**: エクスポートは専用の新規接続で `BEGIN` → 全セル順次実行 → 既定 `ROLLBACK`(`--allow-writes` 時のみ `COMMIT`)。SAVEPOINTは使用しない。CLIは常に新規接続のため「既にトランザクション中」は発生しない(Phase 2の同一プロセス呼び出しにおける扱いは未解決 — design doc 6.2#7)。
2. **エラー2分類**: 非abort系エラー(CatalogException等)は失敗セルとして記録し後続を継続。abort系エラー(ConstraintException等)発生後は**後続セルを実行せず「トランザクション中断のためスキップ」としてHTMLに明示**する。abort判定はエラー捕捉直後のプローブ文(`SELECT 1`)で `TransactionException` を検知する方式。
3. **セル内トランザクション制御文の拒否**: `BEGIN`/`COMMIT`/`ROLLBACK` を含むセルは実行せず失敗扱いにする(COMMITを許すとROLLBACKによる安全策ごと確定されるため)。判定はSQLパースで行う。
4. **タイムアウト**: セル実行を専用スレッドで行い、超過時に `connection.interrupt()` を呼ぶ。interruptによりabort状態になった場合は決定2の処理に合流する(状態遷移の詳細は要検証 — design doc 6.2#9)。interrupt後さらに猶予(既定30秒)を待っても実行スレッドが戻らない場合は、同一接続での後続セル実行が不可能なため、後続セルの実行を断念して取得済み結果で部分HTMLを生成し(未実行セルはスキップ表示)、非ゼロ終了する(2026-07-05追記。CLIプロセスがハングしないことを保証する)。
5. **外部副作用はこのADRの保護範囲外**: ROLLBACKが保護するのは対象DBファイル内のみ。COPY TO等の扱いはADR-006で規定する。

### 検討した代替案
- **abort時にリプレイして継続**(ROLLBACK→新トランザクションで成功済みセルを再実行し、失敗セルを飛ばして継続): 全セルの結果が得られるが、実行時間が最悪2倍以上になり、外部副作用も重複実行される。abort系エラーを含むnotebookは稀と想定されるため、単純で予測可能なスキップ方式を採用。将来オプション(`--replay-on-abort` 等)として追加する余地は残す。
- **セル毎に独立トランザクション**: abort問題は消えるが、セル1のCREATE TABLEをセル2が参照する通常のnotebookが動かなくなるため不採用。

### 影響
- design doc 4.2節のExecutor仕様を本ADRに合わせて全面改訂した(第2回改訂)。
- 6.3の検証結果は自動テスト化し、DuckDBバージョン更新による前提変化を検知する(design doc 8章)。

## ADR-008: 実行コンテキストの解決方針(対象DB・環境再現・対応プラットフォーム)

- ステータス: Accepted(2026-07-05新設)

### コンテキスト
Phase 1の実装に着手すると即座に判断を迫られるにもかかわらず、方針が未決だった3点を決定する。

1. **対象DBの解決失敗時の挙動**: notebook JSONに接続先DBの情報が含まれているかは未調査(design doc 6.2#3)であり、「情報がない」が常態の可能性もある。`--db` も未指定の場合にどうするかが未定義だった。
2. **実行環境の再現範囲**: UIセッションが持っていた状態(ATTACH・拡張ロード・シークレット・変数)をどこまで再現するか、方針(やる/やらない)自体が未決だった。httpfs等を使うnotebookはこれ次第で動作可否が変わる。
3. **Windowsの位置づけ**: ロック中ui.dbのOSコピー可否がWindows未検証(design doc 6.2#8)のまま、サポート対象かどうかが明記されていなかった。

### 決定
1. **対象DB解決の優先順位**: `--db` 明示指定 → notebook JSON内の接続情報 → **`:memory:` フォールバック**。フォールバック時は標準エラー出力とHTML内メタデータに警告を明示する。既存テーブルを参照するセルはCatalogException(非abort系)として失敗表示され後続は継続されるため、ADR-007のエラー処理と自然に整合する。自己完結型notebook(CREATE TABLE→SELECT)はDB指定なしでもエクスポートが完走する。
2. **実行環境の再現**: notebook JSONに接続環境情報が含まれていれば**可能な範囲で再現**する。情報がない要素はDuckDBの既定動作(autoload)に任せ、依存するセルは失敗セルとして扱う。この決定によりnotebook JSONスキーマの実機調査(design doc 6.2#1/#3)がPhase 1実装の**前提タスク**となる。
3. **対応プラットフォーム**: Phase 1は**Linux/macOSのみ正式サポート**。Windowsはコード上排除しないが未検証と明記し(README)、検証はPhase 1.5以降に繰り延べる。CIマトリクスもLinux/macOSのみで構成する。

### 検討した代替案
- **解決失敗時に `--db` 必須エラー**: 挙動は最も予測可能だが、「JSONに情報がない」が常態だった場合に毎回の指定を強いる。フォールバック+警告の方が体験と安全性のバランスが良いと判断。
- **解決失敗時に対話プロンプトでDBパス入力**: CI等の非対話環境で使えず、`--yes` との整合も複雑になるため不採用。
- **環境再現をしない(常に素の接続)**: 実装は最小だが、httpfs・ATTACHを使う一般的なnotebookが常に壊れる。JSONに情報がある場合にまで無視する理由がないため不採用。
- **環境再現用の `--init` スクリプトオプション**: 実装は単純だが責任をユーザーに転嫁する。将来の補完手段としては排除しない(JSON由来の再現で不足するケースの逃げ道)。
- **Windows検証をPhase 1に含める**: CI検証コストが増え、MVPのクリティカルパスでない。ロック中コピーの成否はコードで対処できない可能性もあるため、検証自体を後続フェーズに送る。

### 影響
- design doc 4.2節(Executor)・7章(`--db` の既定動作)・8章(対応プラットフォーム)を本ADRに合わせて改訂した(第3回改訂)。
- notebook JSONスキーマ調査(design doc 6.2#1)がPhase 1実装の前提タスクに昇格した。

### 追記(2026-07-05、notebook JSONスキーマ実機調査)
実機調査によりJSON内のDB情報は名前のみと判明した(6.2#3解決、design doc 6.3#9)。決定1のフォールバック連鎖(`--db` → notebook JSON → `:memory:`)自体は変更なし。DB名から実ファイルへの解決は行わない(名前のみでは解決不能なため)。

### 追記(2026-07-05、Phase 1実装完了に伴う整合性レビュー)
決定2(実行環境の再現)について、Phase 1実装では**未適用**である。DB名のみでは再実行接続への適用セマンティクス(`USE`の適用対象・失敗時の挙動)が未確定のため保留とした(design doc 4.2節参照)。決定2自体(可能な範囲で再現するという方針)は維持する。
