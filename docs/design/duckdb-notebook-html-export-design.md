## 1. 背景・課題

DuckDB UI(MotherDuck製、`duckdb --ui` で起動するnotebook機能)には、notebookをHTMLとしてエクスポートする機能が公式にもコミュニティ拡張にも存在しない。

- notebookの内容は `notebooks` / `notebook_versions` テーブルを持つ内部DB(`<HOME>/.duckdb/extension_data/ui/ui.db`)にJSON形式(`notebook_versions.json`)で保存されている。
- DuckDB UIのフロントエンド実装(TypeScript製)は非公開(`https://ui.duckdb.org` から配信されるクローズドソース)のため、チャート描画などのロジックは参照できない。
- Jupyterの `nbconvert` に相当する「notebookの構造を読み取り、実行結果を含めて静的HTMLとして書き出す」ツールが存在しない。

さらに、外部CLIとして別途実行するだけでなく、**Jupyterの `%%` マジックコマンドのような手軽さで、notebookのSQLセルから直接エクスポートを呼び出せること**を重視したい。

## 2. ゴール

### 2.1 フェーズ分け(MVP / Phase 1.5 / 拡張)

「マジックコマンド」「CLI」「チャート」を同時に満たすのはコストが高いため、フェーズを分ける。**本ドキュメントのGoal/非ゴールは常にこのフェーズを前提とする**(ADR-001〜ADR-007もこのフェーズ分けと整合させてある)。

| フェーズ | スコープ | 対応ADR |
|---|---|---|
| **Phase 1(MVP)** | ①CLIエクスポート(テーブルのみ、チャート非対応) ②再実行結果への安全策(単一トランザクション + ROLLBACK) | ADR-001, ADR-002, ADR-003, ADR-006, ADR-007 |
| **Phase 1.5(マジックコマンド)** | ③notebookセルからのマジックコマンド呼び出し(案A: Python再利用)。**着手の前提条件**: UI稼働中の対象DBファイルロック問題(6.2#6)の解決策が実機検証で確立していること | ADR-005 |
| **Phase 2(拡張)** | ④チャートのクライアントサイド埋め込み(**条件付き** — DuckDB UIにチャート設定の永続化機構が存在しないことを確認済み(6.2#2)のため、UI側が永続化を導入するまで着手不能。ADR-004はDormant) ⑤レンダリングコアのC++移植とコミュニティ拡張化(案B) ⑥CLIもC++コアを呼ぶ形に統合 | ADR-004, ADR-005 |

**第2回改訂でマジックコマンドをPhase 1から分離した理由**: 実機検証(6.3#4)により、DuckDBのDBファイルはあるプロセスがRW接続を保持している間、**別プロセスからは `read_only` ですら開けない**ことが確認された。Phase 1のサブプロセス方式マジックコマンドは「UIがいま開いているDBに対してセルを再実行する」という典型シナリオそのもので動作しないため、解決策(対象DBのスナップショットコピー、UIが内部的に使うHTTPエンドポイントへの相乗り等)の検証が完了するまで独立フェーズとして分離する。Phase 1はCLIのみでMVPとして完結させる。

### 2.2 Goal(達成基準)

- **Goal 1(表・チャートの静的出力)**: DuckDB UIのnotebook(1つ以上のSQLセルで構成)を、実行結果を含む単一の静的HTMLファイルとしてエクスポートできること。
  - Phase 1達成基準: 各セルのSQLと再実行結果のテーブル(先頭N件、4.3節で規定する上限あり)がHTMLに含まれていればよい。チャートはPhase 1のスコープ外(未達成でも良い)。
  - Phase 2達成基準(条件付き — 2.1の④参照): チャートセルについてもクライアントサイドの宣言的埋め込み(ADR-004、現在はDormant)で表現されていること。DuckDB UIがチャート設定の永続化を導入するまでは対象外。
- **Goal 2(マジックコマンド)【Phase 1.5】**: notebook内のSQLセルから、以下のセマンティクスで直接エクスポートを実行できること。
  - 呼び出し形式: `SELECT export_notebook_html(notebook_name VARCHAR, output_path VARCHAR DEFAULT NULL) -> VARCHAR` というスカラー関数。
  - **同期実行**: 呼び出しはエクスポート処理(再実行・レンダリング・ファイル書き込み)が完了するまでブロックする。非同期・バックグラウンド実行はサポートしない(結果ファイルへの参照を即座に返す必要があるため)。
  - **戻り値**: 成功時は生成したHTMLファイルの絶対パス(VARCHAR)を返す。失敗時はSQL例外を送出する(呼び出し元セルはエラー扱いになる。7章のエラー処理方針を参照)。
  - **`'current'` の意味**: `notebook_name` に `'current'` を渡した場合、「この関数を呼び出しているセルが属しているnotebook」を指す。解決方法は、DuckDB UIがセル実行時にそのconnectionへ notebook/セルの識別情報を伝搬している前提で、関数内部がその識別情報を読み取って解決する(詳細はADR-005参照。DuckDB UI側がこの情報を実際に伝搬しているかは非公開仕様のため実機検証が必要 — 6.2「要調査事項」参照)。**その情報が取得できない場合(例: 通常の `duckdb` CLIセッションやUI以外から呼んだ場合)は `'current'` はサポートせず、明示的にnotebook名を渡すことを要求する明確なエラーを返す**。これによりGoal 2は「'current'が使えれば理想だが、使えなくても明示name指定で必ず動く」という形で常に達成可能にする。
  - 実装方式はADR-005の案A(Pythonサブプロセス方式)を採用する。ただしGoal 2全体が、6.2#6(対象DBロック問題の解決策)の検証完了を**着手条件**とする。
- **Goal 3(CLI・エンジン共有)**: 単体のCLIとしても実行できること(`duckdb-nb-export <notebook-name> -o out.html` のようなイメージ、詳細は7章)。
  - 「CLIとマジックコマンドは同じレンダリングエンジンを共有する」とは、**同一フェーズ内で常に同一の実装コードパスを通る**ことを指す。
    - Phase 1.5: マジックコマンド(DuckDB拡張の薄いシム)は、CLIと全く同じPythonレンダラー(ADR-001)をサブプロセスとして呼び出す。したがってCLIとマジックコマンドは文字通り同じPythonコードを実行する。
    - Phase 2: レンダリングコアをC++へ移植する(ADR-005案B)にあたり、**CLI側もこのC++コアを呼ぶ形に作り直す**(例: CLIをC++コアにリンクした薄いバイナリにする、またはDuckDB本体+拡張をCLIとして起動する形にする)。これにより、コアの実装言語がPython→C++に変わっても「CLIとマジックコマンドが同じエンジンを共有する」というGoal 3自体は両フェーズで矛盾なく成立する。
  - この整理により、ADR-005で将来C++再実装(案B)を選んでも、Goal 3とは矛盾しない(1回目レビュー指摘の解消)。

## 3. 非ゴール

- notebookの構造そのものの編集・作成機能(読み取り専用)。ただし「読み取り専用」の正確な範囲は以下のとおり。
  - notebook定義(ui.db)への書き込みは一切行わない。
  - 対象DBへの再実行による変更は、既定では単一トランザクション + ROLLBACK(4.2節・ADR-007)により**対象DBファイル内に関しては**呼び出し前の状態に戻す。
  - **ただしROLLBACKで巻き戻るのは対象DBファイル内の変更のみ**である。`COPY ... TO` によるファイル書き出し、ATTACHした別DBへの書き込み、httpfs経由のリモート書き込み等の外部副作用はロールバックされない(実機確認済み、6.3#6)。これらの扱いは5章・ADR-006で規定する(既定は許可 + 実行前確認プロンプト、`--no-external-access` でオプトアウト)。
- DuckDB UI自体の代替UIを作ること。
- リアルタイム/インタラクティブなHTML、すなわち**エクスポート後にサーバー側で新たにクエリを再実行する機能**の提供。あくまで実行時点のスナップショットの静的出力である。
  - 補足: ADR-004でいう「インタラクティブなチャート」は、Vega-Lite等がクライアントサイドJSのみで提供するホバー・ズームといった**見た目上の操作性**を指し、新規クエリ実行やサーバーとの通信を伴わない。上記の非ゴール(サーバー側再実行なし)とは矛盾しない別次元の話である。
- チャート機能の完全な再現(DuckDB UIのチャート仕様が非公開のため、完全互換は保証しない)。Phase 1ではチャート対応自体を行わない(2.2節参照)。
- Phase 1.5における、`:memory:` 等の非永続(オンメモリ)DB上で開かれたnotebookに対するマジックコマンド機能。Phase 1.5のマジックコマンドはサブプロセス(別プロセス)方式のため、呼び出し元セッションのインメモリ状態を共有できない(4.2節・ADR-005参照)。Phase 2でレンダリングコアが同一プロセス内(DuckDB拡張)で完結すればこの制約は原理的に解消されうるが、Phase 2で解消することを本ドキュメントで確約するものではない。
- notebookを共有・配布する際のアクセス制御・認可(誰がどのnotebookをエクスポートしてよいか)の設計。これはDuckDB UI本体のアクセス制御の範囲であり本ツールでは扱わない。

## 4. 全体アーキテクチャ

nbconvertと同様の「読む→実行→描画」のパイプラインを核としつつ、それを**SQLから呼び出せるエントリポイント**(Phase 1.5以降)でラップする2階建て構成にする。

```
                      notebookのSQLセルから直接呼び出し (Phase 1.5以降)
                                  |
                                  v
                  [SQL callable entry point] (DuckDB拡張 / 関数)
                                  |
                                  v (Phase1.5: subprocess橋渡し / Phase2: 同一プロセス内呼び出し)
[ui.db] --(1 Reader)--> [Notebook/Cell モデル] --(2 Executor)--> [セルごとの実行結果]
                                                                        |
                                                              (3 Renderer)
                                                                        v
                                                                  [静的HTML]
```

### (0) SQL callable entry point層 — 「マジックコマンド」の正体【Phase 1.5以降】
- DuckDBの素のSQLには、外部プロセス起動やファイル書き出しを行う手段が標準では無いため、これは**DuckDBの拡張機能(スカラー関数)として実装する**必要がある。
- 関数シグネチャ・戻り値・`'current'` の意味はGoal 2(2.2節)で確定済み。
- Phase 1.5では内部で (1)〜(3) のPythonエンジンをサブプロセスとして呼び出す(ADR-005案A)。Phase 2ではC++コア(ADR-005案B)を同一プロセス内で直接呼び出す。
- **配布上の注意**: サブプロセスを起動する拡張はコミュニティ拡張リポジトリの審査を通らない可能性が高く、Phase 1.5では unsigned 拡張としての配布(利用者側は `allow_unsigned_extensions` を付けてUIを起動する必要がある)を前提とする。「INSTALLしたら動く」体験になるのはPhase 2(案B)以降である(ADR-005)。

### (1) Reader層 — notebook定義の取得
- 対象DBは `ui.db` であり、DuckDB UI稼働中はこのファイルに対して排他ロックがかかる(既知の制約。GitHub Issue #17158)。実機検証(6.3#4)により、ロック保持中は別プロセスから `read_only` でも開けないことを確認済みのため、**コピー方式が唯一の現実的な選択肢**である(詳細はADR-002)。
  - CLI単体実行時: 既定では **ui.dbのスナップショットコピーを作成してからコピーを読む**(UIを起動したまま実行可能にするため)。コピーはOSファイルコピーで良く、DuckDB自体のロックとは競合しない(Linuxで確認済み。Windowsでの可否は要調査 — 6.2#8)。UIを閉じてから実行する `--require-ui-closed` モードも提供する。
  - **WALファイルも必ずセットでコピーする**: 実機検証(6.3#5)で、直近の変更が `ui.db.wal` にのみ存在し、`ui.db` 本体だけをコピーすると最新の編集が見えないことを確認した。コピーは `ui.db` + `ui.db.wal`(存在する場合)の2ファイルをセットで行う。2ファイルのコピーはアトミックではないため、コピー後にnotebook JSONの読み取り検証を行い、不整合が疑われる場合(DBが開けない・JSONパース失敗等)は短い待機(既定0.5秒)を挟んで既定3回までリトライする。それでも失敗する場合は「UIが起動中のため読み取りに失敗した。`--require-ui-closed` で再試行するか、少し時間をおいて再実行してください」という明確なエラーを返す。**ソースの `ui.db` ファイル自体が存在しない場合(第7回改訂)**はこのリトライ・「UIが起動中かもしれない」という文言の対象外とし、即座に「`ui.db` が見つからない。`--ui-db` を指定するか、一度DuckDB UIを起動して作成してください」という趣旨の明確なエラーを返す(存在しないファイルに対して「UIが起動中かもしれない」と誘導するのは誤りであるため)。
  - ui.dbのパスは既定 `<HOME>/.duckdb/extension_data/ui/ui.db` とし、`--ui-db` オプションで上書き可能にする(Windows・カスタム環境向け)。
  - マジックコマンド実行時(Phase 1.5、別プロセスへsubprocess委譲): 実体はCLI単体実行と同じ「コピーしてから読む」方式に委譲する(UI稼働中でも動作する)。
  - マジックコマンド実行時(Phase 2、同一プロセス内): UIサーバーと同一プロセスであるため、新たにファイルを開き直して二重ロックを取得する必要がない。UIサーバーが保持している既存のDB接続/インスタンスを再利用してnotebook定義を取得する。
- 取得したJSONは **Pydanticモデル**(Phase 1)でパースする。未知のフィールドを許容し、将来のスキーマ変更で壊れないようにする。パース対象は実機調査(6.3#9)で確認した**保存形式v3**(`notebookSerializationFormat: 3`)であり、notebook本体側のバージョンは `notebook_versions.version`(整数)、最新版は `expires IS NULL` で判定する。
- notebook不在時: 「指定したnotebook名が見つからない」ことを明確なエラーメッセージ(存在するnotebook名の一覧を添えることが望ましい)として返す。CLIは非ゼロ終了コード、マジックコマンドはSQL例外とする(7章)。
- **同名notebookの衝突**: notebook名で複数のnotebookが一致した場合は曖昧さを解決せず、候補(名前と内部ID・更新日時)を一覧表示してエラーとする。バージョンは既定で最新を対象とし、`--nb-version` で特定バージョンを指定可能にする。**`--notebook-id <id>` による明示指定で回避可能**(第7回改訂)。`--notebook-id` を指定した場合は名前ではなくIDで一意に解決し、位置引数の notebook名は省略できる(両方与えられ、かつ一致しない場合もIDを優先する)。該当IDが存在しない場合は既存ID一覧を添えた `NotebookNotFoundError` とする。

### (2) Executor層 — セルの実行
- notebookのJSONはSQL定義を持つのみで実行結果は保持していない可能性が高いという前提のもと、エクスポート時に対象DBに対して各セルのSQLを**再実行**して結果を取得する(nbconvertの `--execute` 相当)。
- **実行対象DBの解決(ADR-008)**: `--db` 指定があればそれを使う。未指定時はnotebook JSON内の接続情報からの解決を試みる(スキーマ調査 6.2#3)。**どちらでも解決できない場合は `:memory:` に対して実行**し、その旨を標準エラー出力とHTML内のメタデータに警告として明示する。既存テーブルを参照するセルはCatalogException(非abort系エラー)として失敗表示され、後続セルの実行は継続される(ADR-007のエラー処理と整合)。**実機調査完了(6.3#9)**: JSON内の `currentDatabase`/`cells[].useDatabase` はDB名のみでファイルパス・DSNを含まないため、この解決手段は事実上使えず、`:memory:` フォールバックが実質的な既定経路になる。
- **実行環境の再現(ADR-008)**: notebook JSONにATTACH・拡張ロード・シークレット・変数などの接続環境情報が含まれている場合は、可能な範囲で再現してから再実行する。情報がない要素についてはDuckDBの既定動作(httpfs等のautoload)に任せ、再現されない環境に依存するセルは失敗セルとして扱う。**実機調査完了(6.3#9)**: 保存形式v3にはATTACH・拡張ロード・シークレット・変数等の接続環境情報を保持するキーは存在しないため、再現できる情報は事実上なく、素の接続+autoload任せが常態になる。**適用セマンティクス確定・実装済み(2026-07-05)**: 保存JSONが持つDB名(`currentDatabase`/`useDatabase`)は**ベストエフォートの `USE`** で適用する。notebookレベルの `currentDatabase` はBEGIN直後に1回、セルレベルの `useDatabase` は当該セルの実行直前に発行する。名前が解決できる(同名カタログがATTACH済み — `--db` 指定または先行セルのATTACH)場合のみ切り替わり、失敗時は `CatalogException` を捕捉して警告(stderr+HTMLメタデータ)を出し、現在のデータベースのまま続行する(失敗した `USE` はトランザクションをabortさせないことを実機確認済み)。同一の解決不能名について**警告は1回のみに重複排除するが、`USE` の試行自体は毎セル行う**(先行セルのATTACHにより後から解決可能になり得るため)。また、タイムアウトによるabort復旧(ROLLBACK+BEGIN)でトランザクション内ATTACH由来のカタログが失われ、デフォルトデータベースが無効になっている場合は、プライマリカタログへ警告付きで復元する。テスト設計書UT-X-021/UT-X-022/UT-X-023/UT-X-024が対応。
- **トランザクション戦略(ADR-007)**: notebookのセルにはDDL/DMLが含まれうるため、再実行によって対象DBが意図せず変更される(重複INSERT・DROPなど)リスクがある。これを避けるため、エクスポート実行は既定で以下のように行う。
  1. エクスポート用の**新規接続**を開き、`BEGIN TRANSACTION` する。CLIは常に自前の接続を使うため「呼び出し時点で既にトランザクション中」という状況は発生しない。
     - **注**: DuckDBは `SAVEPOINT` / ネストトランザクションをサポートしない(実機確認済み、6.3#1)。第1回改訂版にあった「既にトランザクション中であればSAVEPOINTを発行する」という記述は実装不可能なため削除した。Phase 2(同一プロセス呼び出し)で呼び出し元が既にトランザクション中の場合の扱いは未解決論点として6.2#7に残す。
  2. notebook内の全セルを順に実行する。
  3. 全セル実行後、既定では必ず `ROLLBACK` して対象DBの状態を呼び出し前に戻す。結果セット自体はロールバック前に取得済みのためHTMLには反映される。DDL(CREATE/DROP TABLE等)もROLLBACKで完全に巻き戻ることを実機確認済み(6.3#2)。
  4. セルのDML/DDLをそのまま対象DBへ反映したい場合のみ、CLIは `--allow-writes` オプション(最後にROLLBACKの代わりにCOMMITする)を要求する。既定は「反映しない(ROLLBACK)」。
- **セル内のトランザクション制御文の拒否**: セルSQLに `BEGIN` / `COMMIT` / `ROLLBACK` が含まれる場合、そのセルは**実行せず失敗扱い**とし、HTMLに「セル内のトランザクション制御文はエクスポート時にサポートされない」旨を明示する。実機確認(6.3#3)のとおりトランザクション内での `BEGIN` はエラーになり、`COMMIT` を許すと安全策(ROLLBACK)ごと確定されてしまうため。判定はSQLパース(`duckdb.extract_statements` 相当)で行う。
- **エラー処理方針(ADR-007)**: 既定は「あるセルの実行が失敗しても中断せず、後続セルの実行を継続する」(nbconvertの `--allow-errors` 相当)。ただし実機検証(6.3#3)により、DuckDBではエラー種別によってトランザクションへの影響が異なることが判明したため、以下の2分類で扱う。
  - **非abort系エラー**(CatalogException = テーブル不在、BinderException等): トランザクションは有効なまま継続できる。失敗セルにエラーメッセージを表示し、後続セルの実行を継続する。
  - **abort系エラー**(ConstraintException等、トランザクション全体が無効化されるもの): 以降の文はすべて `Current transaction is aborted` で失敗するため、**後続セルは実行せず「トランザクション中断のためスキップ」としてHTMLに明示表示**する(SAVEPOINTが存在しないため文単位の復旧はできない)。
  - abort判定の実装: セル実行エラーを捕捉した直後に軽量なプローブ文(`SELECT 1`)を実行し、`TransactionException` が返る場合をabort状態と判定する。
  - 厳密モードとして `--stop-on-error` を提供し、指定時は最初の失敗(種別を問わず)で処理を中断する。
- **タイムアウト**: セル単位で既定300秒のタイムアウトを設ける(`--cell-timeout` で変更可能)。実装方式は「セル実行を専用スレッドで行い、タイムアウト時に監視側から `connection.interrupt()` を呼ぶ」。interrupt後も同一接続が再利用可能であることは実機確認済み(6.3#7)。超過したセルは失敗として扱い、interruptによってトランザクションはabort状態になる(実測確定 — 6.2#9、6.3#10、AT-011)ため、上記abort系エラーの処理に合流する(実装はプローブ後に `ROLLBACK`+`BEGIN` でトランザクションを再開して後続セルを継続する)。
  - **interruptが効かない場合のフォールバック(第4回改訂)**: interrupt後さらに猶予(既定30秒)を待っても実行スレッドが戻らない場合(拡張内の長時間ブロック等、稀だが起こりうる)、同一接続での後続セル実行は不可能なため、後続セルの実行を断念し、その時点までに取得済みの結果で**部分HTMLを生成**(未実行セルはスキップ表示)して非ゼロ終了する(終了コード2)。CLIプロセスがハングしないことを保証する。
- **外部副作用の扱い(ADR-006)**: `COPY ... TO` のファイル書き出しはトランザクション内で実行してROLLBACKしても**ファイルが残る**ことを実機確認済み(6.3#6)。同様にATTACHした別DBへの書き込み、httpfs経由のリモート書き込み、`INSTALL`/`LOAD` もロールバックされない。既定ではこれらを遮断しない(`read_parquet`/`read_csv` 等の外部読み込みはDuckDBの中核ユースケースであり、遮断すると一般的なnotebookが動かなくなるため)。防波堤は5章の実行前確認プロンプト(セルSQL一覧の目視)とし、加えて `--no-external-access` オプション(`SET enable_external_access = false` で実行。外部ファイルの読み込みも無効になる点に注意)を提供する。
- notebook自身から呼び出す場合、「今まさに開かれているDB接続」を使い回せるかはフェーズで異なる(2.2節Goal 3参照)。Phase 1.5(別プロセス)では同一の**ディスクファイル**を指す新規接続を開いて再実行する必要があるが、UI稼働中はロックにより開けない(6.3#4)— この解決策の確立がPhase 1.5の着手条件である(6.2#6)。Phase 2(同一プロセス)では呼び出し元の接続・トランザクションコンテキストをそのまま利用できる。

### (3) Renderer層 — HTML生成
- **Phase 1**: **Jinja2** でHTMLテンプレートを組み立て、**Pygments** でSQLのシンタックスハイライトを行う(ADR-001)。CSS(Pygmentsのスタイル含む)はすべてHTMLにインライン埋め込みし、外部参照のない単一ファイルとする。
- **デザイン方針(第4回改訂)**: 独自のミニマルCSS(システムフォントスタック)とし、`prefers-color-scheme` によるダークモード自動対応を行う。nbconvertやDuckDB UIの見た目の模倣はしない(非公開デザインへの追従コストを避け、依存なく保守できるようにするため)。
- **出力言語(第4回改訂)**: 生成HTML内の注記文言("Charts are not supported in Phase 1"、"Skipped: transaction aborted" 等)およびCLIメッセージ・エラーメッセージは**英語**とする(PyPIでの一般公開を前提とするため)。設計ドキュメント類は日本語のままで良い。
- **Phase 2**: レンダリングコアをC++へ移植する(ADR-005)。テンプレートはヘッダオンリーC++テンプレートエンジン(inja または Jinja2Cpp)を使用し、SQLシンタックスハイライトはDuckDB自身の内部トークナイザ(`Parser::Tokenize()` / `duckdb_keywords()`)またはコミュニティ拡張 `poached` の `tokenize_sql()` を流用する。ゼロからのフルスクラッチではないため、Phase 2の実装コストは「最も高い」ではなく「中コスト」と見積もる(ADR-005参照)。チャート用JS(Vega-Lite等)も「単一静的HTML」ゴールを優先してインライン埋め込みを既定とする(サイズ影響はPhase 2着手時に評価)。
- クエリ結果は `<table>` に変換する。**表示件数上限は既定1,000行**とし(`--max-rows` で変更可能)、結果の取得は**上限+1行のみ**(fetchmany相当)とする。上限を超えた場合は「先頭1,000行を表示(1,000行超。総件数は未計測)」の形で表示する。正確な総件数の取得は全件fetch(メモリリスク)またはCOUNTクエリの再実行(実行時間・副作用の二重実行リスク)を要するため**行わない**(第3回改訂で「総件数を表示」から緩和)。全件が必要な場合はCSV等への別出力を将来の拡張候補とする(本フェーズでは対象外)。
- **結果表示の詳細規定**:
  - 1セルに複数ステートメントが含まれる場合: すべて順に実行し、**最後のステートメントの結果セット**を表示する(第3回改訂で確定。duckdb CLIをはじめ多くのSQLツールの慣習と一致するため、DuckDB UIの実際の表示仕様が異なると判明した場合でも本仕様を優先する)。
  - **チャートセル(Phase 1)**: SQLは通常のセルと同様に再実行し、結果を**通常のテーブルとして代替表示**した上で「チャート表示はPhase 1では非対応(テーブルで代替表示)」と注記する(ADR-004)。実行スキップにはしない(後続セルがチャートセルのSQLに依存するケースを壊さないため)。
  - 結果セットを返さない文(DDL等): 「OK」等の完了表示。DML: 影響行数を表示する。
  - NULLは `NULL` と明示表示(空文字列と区別)。ネスト型(STRUCT/LIST/MAP)はDuckDBの文字列表現で表示。BLOBは値そのものではなくサイズ表示。極端に長い文字列値は**既定500文字**で切り詰めて全長を注記する(第4回改訂で具体化)。
  - `CREATE SECRET` 文は値をマスクして表示する(5章)。
- **出力メタデータ**: 生成したHTMLには以下のメタデータをフッターまたは`<head>`のメタタグとして必ず含める。
  - エクスポート実行日時(UTC, ISO8601)
  - DuckDBのバージョン(`PRAGMA version` 相当)
  - 対象notebookのバージョン識別子(`notebook_versions` のバージョンID)
  - 本ツール自体のバージョン

## 5. セキュリティ考慮事項

出力先パス制御・XSS対策・信頼境界・シークレット漏洩・外部副作用の5点を本節で決定する(詳細な決定理由はADR-006)。

- **出力パス制御(パストラバーサル対策)**: `output_path` / `-o` に渡された値は必ず絶対パスへ正規化(symlink解決含む)した上で、既定の許可ベースディレクトリ(CLI: カレントディレクトリまたは `--output-dir` で指定したディレクトリ、マジックコマンド: DuckDB起動時の作業ディレクトリ配下)の外を指す場合は拒否する。`..` を含むパスの単純な文字列拒否ではなく、正規化後のパス比較で判定する。ベースディレクトリ外への出力が必要な場合は明示的に許可ベースディレクトリ自体を変更するオプションで対応し、暗黙の脱出は許可しない。
- **XSS対策**: Jinja2/inja双方とも `autoescape` を既定で有効にする。SQL文字列・クエリ結果のセル値は常にエスケープ済みとしてテンプレートに渡し、生HTMLとしての注入は行わない(将来Markdownセル等をサポートする場合も、限定的なサニタイズ済みサブセットのみ許可する設計とし、本フェーズのスコープには含めない)。
- **シークレットのマスキング**: エクスポートHTMLは共有・配布されることが前提の成果物であるため、セルSQLに含まれる `CREATE SECRET` 文を検出し、**すべてのパラメータ値**(TYPE・PROVIDER・SCOPEのような構造的要素を除く)を一律 `***` にマスクしてからHTMLへ出力する(第4回改訂で確定。KEY_ID/SECRET/TOKEN等の既知の秘密キー名のみをマスクする案は、プロバイダ追加時のリスト保守漏れがそのまま漏えいになるため不採用。REGION等の非秘密値も隠れるが実害は小さい)。任意の文字列リテラルに埋め込まれたパスワード等の網羅的検出は行わない(過剰検出・誤検出を避けるため)。**残余リスク**: `CREATE SECRET` 以外の形でSQL本文に埋め込まれた秘密情報はそのままHTMLに含まれる。実行前確認プロンプトでのSQL一覧の目視が最後の防波堤であることをユーザー向けドキュメントに明記する。
- **外部副作用の信頼境界**: ROLLBACKによる保護範囲は**対象DBファイル内の変更のみ**である。`COPY ... TO`(実機確認済み、6.3#6)・ATTACH先DBへの書き込み・httpfs経由のリモート書き込み・`INSTALL`/`LOAD` は巻き戻らない。既定でこれらを遮断しない理由と `--no-external-access` によるオプトアウトは4.2節のとおり。
- **信頼境界(他者作成notebookの再実行)**: 本ツールは「呼び出し元が既に持っている権限」の範囲でSQLを再実行するだけであり、新たな権限昇格は発生しない。したがって主なリスクは権限昇格ではなく「意図しないDDL/DML・外部副作用の実行」であり、DBファイル内についてはROLLBACK既定(ADR-007)で緩和される。加えて、他者から共有されたnotebookをCLIでエクスポートする場合(＝実行者がnotebookの作者と異なりうるケース)は、そのnotebookが何をするか実行前に把握していない可能性があるため、CLIは実行前にセル一覧(SQL本文)をコンソールへ表示し、`--yes` を明示しない限り確認プロンプトを挟む。マジックコマンド経由(notebookが開かれているセッション内で自分自身のnotebookをエクスポートする場合)は実行者=notebookの実行者であることが前提のため、確認プロンプトは不要とする。

### 6.2 要調査事項(投機的・実機検証が必要)

| # | 論点 | 備考 |
|---|---|---|
| 1 | notebook JSONの正確なスキーマ | **完了(2026-07-05)**。実機調査によりui.dbの4テーブルDDL・notebook JSON保存形式v3の構造を確認した。詳細は6.3#9を参照 |
| 2 | チャートセルの表現方法 | **完了(2026-07-05、恒久的制限として確定)** — フロントエンドバンドル(ui拡張 0dfdf34 / hatchling.11eef943)の全文再調査により、チャート設定はどこにも永続化されていないことを確認した。根拠: ①`StoredNotebookCellValidator` のキーは `query`/`cellId`/`useDatabase`/`isActive`/`runMode` の5つのみ ②バンドルが発行する `CREATE TABLE` は既知の4テーブル(`notebooks`/`notebook_versions`/`current_notebook_id`/`has_onboarded`)のみで、チャート設定用テーブルは存在しない ③バンドル内の "chart" 関連コードはすべてMotherDuckクラウド機能(Dives)・クエリ実行アクティビティ表示・d3色定義であり、notebookセルのチャート設定を扱うコードは存在しない。すなわちローカルDuckDB UIにはセルのチャート設定を保存する機構自体がなく、「エクスポートで再現すべき保存済みチャート設定」はそもそも存在しない。ADR-004のPhase 2チャート再現は、将来DuckDB UIがチャート設定の永続化を導入した場合にのみ再開する(ADR-004追記参照) |
| 3 | 実行対象DBの解決方法 | **完了(2026-07-05)** — JSONにはDB名(`currentDatabase`/`cells[].useDatabase`)のみが含まれ、ファイルパス・DSNは存在しないことを確認した(6.3#9)。したがってADR-008の `:memory:` フォールバックが実質的な既定経路になる |
| 4 | `'current'` 解決のためのセッションコンテキスト伝搬 | DuckDB UIがセル実行時にconnectionへnotebook/セルの識別情報を伝搬しているか(非公開仕様)を実機調査する。伝搬がない場合は2.2節で定義したフォールバック(明示name必須・エラー)を採用する |
| 5 | 同一プロセス内でのui.db再アクセス手段 | Phase 2でUIサーバーと同一プロセス内から安全にDBインスタンスを共有する具体的なAPI/手順の実機検証 |
| 6 | **UI稼働中の対象DBロック問題の解決策(Phase 1.5の着手条件)** | UI稼働中、対象DBは別プロセスから一切開けない(6.3#4)。**調査は(a)を優先すると決定(第3回改訂)**: (a)対象DBもスナップショットコピーする — ui.dbで確立済みのWALセットコピー方式の延長で技術リスクが低く、非公開仕様に依存しない。残る評価軸はDBサイズ・整合性・コスト。(b)UIが内部的に使うクエリ実行HTTPエンドポイントへの相乗りは、(a)が巨大DBでコスト面から不成立の場合のフォールバック候補として残す(ADR-005) |
| 7 | Phase 2で呼び出し元が既にトランザクション中の場合の安全策 | SAVEPOINTが存在しないため、ネスト相当の巻き戻し手段がない。エラーにする/呼び出し元トランザクションに相乗りする等の方針を実機検証を踏まえて決定 |
| 8 | Windowsでのロック中ファイルコピー可否 | Linuxではロック保持中のOSコピーが可能なことを確認済み(6.3#5)。Windowsは排他ロックによりコピー自体が失敗する可能性があり要検証。**Phase 1はLinux/macOSのみ正式対応と決定した(6.1#14、ADR-008)ため、本調査はPhase 1.5以降に繰り延べ** |
| 9 | interrupt時のトランザクション状態遷移 | **完了(2026-07-05)** — 実測(duckdb 1.5.4)により、**明示的トランザクション内で実行中のクエリをinterruptするとトランザクションはabort状態になる**(後続文は `TransactionException: Current transaction is aborted (please ROLLBACK)`、事前の書き込み有無を問わず同じ。`ROLLBACK` で接続は回復する)ことを確認した(6.3#10)。abort系エラー処理(6.1#3)への合流が正しい経路であり、実装のプローブ方式(`SELECT 1`)+ROLLBACK再開はこの挙動を正しく処理している。AT-011として恒久テスト化済み。なお6.3#7(interrupt後も同一接続を再利用可)はautocommit(トランザクション外)の観測であり矛盾しない |
| 10 | duckdbパッケージとユーザー環境のstorage version整合 | ツール同梱の `duckdb` Pythonパッケージが古いと、より新しいDuckDBで作られたui.db・対象DBを開けない。依存バージョンの追従方針とエラーメッセージ設計(8章)。**実装済み・検証済み(2026-07-05)**: `StorageVersionMismatchError`+更新を促す英語メッセージ+終了コード4。実ミスマッチフィクスチャ(devビルド製、version number 999)によりUT-R-013で実際にこの経路が発火することを確認済み。フィクスチャ運用はテスト設計書2.4節を参照 |

### 6.3 実機検証済み事項(2026-07-05、duckdb 1.5.4 / Linuxで検証)

第2回改訂の根拠となった検証結果。DuckDBのバージョン更新時に前提が変わっていないか、これらは自動テストとして保守する(8章)。

| # | 検証項目 | 結果 |
|---|---|---|
| 1 | SAVEPOINTのサポート | **非対応**(`ParserException`)。ネストトランザクションも不可 |
| 2 | DDL(CREATE/DROP TABLE)のROLLBACK巻き戻り | **完全に巻き戻る**(旧6.2#6は解決) |
| 3 | トランザクション内エラー後の継続可否 | エラー種別による。CatalogException後は継続可。**ConstraintException後はトランザクション全体がabort**され以降の文はすべて失敗 |
| 4 | RW接続保持中のDBファイルへの別プロセスアクセス | **read_only含め一切開けない**(`Conflicting lock is held`) |
| 5 | ロック保持中のOSファイルコピー | Linuxでは可能。ただし**直近の変更はWALにのみ存在し、本体だけのコピーでは見えない**ことを確認(WALセットコピーが必須) |
| 6 | トランザクション内の `COPY TO` とROLLBACK | **書き出したファイルは残る**(外部副作用は巻き戻らない) |
| 7 | `connection.interrupt()` によるクエリ中断 | 可能(`InterruptException`)。中断後も同一接続を再利用できる |
| 8 | トランザクション内の ATTACH / CHECKPOINT / SET | いずれも実行自体は可能(トランザクション内BEGINのみエラー) |
| 9 | notebook JSONスキーマ・ui.db非公式スキーマ・HTTP API(検証環境: duckdb v1.5.4 / ui拡張 164-0dfdf3461f) | `ui.db` は `notebooks` / `notebook_versions` / `current_notebook_id` / `has_onboarded` の4テーブル(外部キー制約なし、DDLは下記)。notebook JSONは**保存形式v3**(`notebookSerializationFormat: 3`)で、トップレベルは `cells` / `currentDatabase` / `viewMode` / `version`。各セルは `cells[].query`(SQL本文)・`cellId` を持ち、**`cell_type`/`cellType` に相当するキーは存在しない**(チャートセルの保存表現自体がv3にない)。`currentDatabase` および `cells[].useDatabase` は**DB名のみ**の文字列で、ファイルパス・DSN等の接続情報は含まれない。最新版の判定は `notebook_versions.expires IS NULL`(旧版は失効時刻が入り期限切れ後に削除)。HTTP API(`/ddb/run` 等)はいずれも401 Unauthorizedで、notebook CRUD専用のRESTエンドポイントは存在せず、UIは `/ddb/run` 経由のSQLで `ui.db`(ATTACH名 `_duckdb_ui`)を直接操作している。詳細はスキーマ実機調査記録を参照 |
| 10 | 明示的トランザクション内でのinterrupt(2026-07-05追加、duckdb 1.5.4) | **トランザクションはabort状態になる**。interruptされたクエリは `InterruptException`、以降の文は `TransactionException`(Current transaction is aborted)で失敗する。事前の書き込み有無(read-onlyトランザクションか否か)を問わず同じ。`ROLLBACK` 後は同一接続を再利用できる。AT-011で恒久ガード |

`ui.db` のDDL(フロントエンドバンドルから抽出、実DBのPRAGMAで検証済み):
```sql
CREATE TABLE IF NOT EXISTS notebooks(
  id UUID NOT NULL PRIMARY KEY,
  name VARCHAR NOT NULL,
  created TIMESTAMP NOT NULL
);
CREATE TABLE IF NOT EXISTS notebook_versions(
  notebook_id UUID NOT NULL,
  version INTEGER NOT NULL,
  title VARCHAR NOT NULL,
  json VARCHAR NOT NULL,
  created TIMESTAMP NOT NULL,
  expires TIMESTAMP,
  PRIMARY KEY (notebook_id, version)
);
CREATE TABLE IF NOT EXISTS current_notebook_id(id UUID NOT NULL);
CREATE TABLE IF NOT EXISTS has_onboarded AS SELECT false AS has_onboarded;
```

notebook JSON(保存形式v3)の例:
```json
{
  "notebookSerializationFormat": 3,
  "cells": [
    {"query": "select 1;", "cellId": 1, "useDatabase": "memory", "isActive": true, "runMode": "default"}
  ],
  "currentDatabase": "memory",
  "viewMode": {"mode": "default"},
  "version": 1
}
```

## 7. CLIインターフェース仕様

コマンド名は `duckdb-nb-export` で確定(第4回改訂)。PyPIパッケージ名もコマンド名と同一の `duckdb-nb-export` に統一する(2026-07-05、実装時に統一を決定。第4回改訂時点の「パッケージ名=検索用・コマンド名=短縮用」という役割分担案は採用しなかった)。

```
duckdb-nb-export <notebook-name> [options]
duckdb-nb-export <notebook-name> --list-versions
duckdb-nb-export --list

必須引数:
  <notebook-name>          対象notebookの名前(位置引数。--list指定時、または--notebook-id
                           指定時は省略可。第7回改訂)

主なオプション:
  -o, --output <path>      出力HTMLパス(既定: ./<notebook-name>.html。notebook名にファイル名として
                           不正な文字・空白が含まれる場合は `_` に置換し、置換後の名前を警告表示する。
                           出力先が既に存在する場合は `<name>-1.html` のような連番を付与して回避する)
  --output-dir <dir>       出力先として許可するベースディレクトリ(既定: カレントディレクトリ)
  --db <path>              再実行対象のDBファイル(既定: notebook JSON内の情報から解決を試み、
                           解決できない場合は :memory: で実行し警告を表示する。ADR-008・4.2節)
  --ui-db <path>           ui.dbのパス(既定: <HOME>/.duckdb/extension_data/ui/ui.db)
  --nb-version <id>        エクスポートするnotebookバージョンID(既定: 最新)
  --notebook-id <id>       notebook IDを指定して名前ではなくIDで一意に解決する(同名notebookが
                           複数存在する場合の脱出路。第7回改訂)。指定時は位置引数の
                           <notebook-name> を省略可能で、両方与えられ不一致の場合もID側を
                           優先する。--list-versions と併用可。該当IDが無い場合は既存ID一覧を
                           添えたエラーとする
  --list                   存在するnotebook一覧(名前・ID・更新日時)を表示して終了
  --list-versions          指定したnotebookのバージョン一覧(ID・作成日時)を表示して終了
                           (--nb-version に渡すIDの発見手段)
  --max-rows <N>           セルごとの表示行数上限(既定: 1000)
  --cell-timeout <sec>     セル単位の実行タイムアウト秒数(既定: 300)
  --stop-on-error          セル実行失敗時に処理を中断する(既定は継続。abort系エラー時の挙動は4.2節)
  --allow-writes           再実行によるDDL/DMLを対象DBへ実際に反映する(既定はROLLBACKにより反映しない)
  --no-external-access     enable_external_access=false で実行し、COPY TO等の外部副作用を遮断する
                           (外部ファイルの読み込みも無効になる点に注意)
  --require-ui-closed      ui.dbをコピーせず直接開く(UIが起動していないことを前提とする厳密モード)
  --yes                    実行前確認プロンプトをスキップする。stdinがTTYでない非対話環境
                           (パイプ・CI等)では、確認が必要な場面で --yes がない場合は
                           プロンプトを出さずエラー終了する(終了コード5)

終了コード:
  0  成功(--stop-on-error未指定でセル失敗を含んだまま完走した場合も0。失敗はHTML内と
     標準エラー出力のサマリで報告する)
  1  notebookが見つからない(同名複数一致による曖昧エラーを含む)
  2  セル実行エラー(--stop-on-error指定時に中断した場合、またはタイムアウト後のinterruptが
     猶予内に効かず全体中断した場合。後者は部分HTMLを生成した上でこのコードを返す)
  3  出力パスが許可ベースディレクトリ外(セキュリティ検査で拒否)
  4  ui.dbへのアクセスに失敗(ロック・破損・storage version不整合等。その他の予期しない
     エクスポート内部エラー(ExporterErrorの包括的な捕捉によるもの)もこのコードを返す)
  5  実行前確認プロンプトでユーザーが実行を拒否した(非対話環境で確認が必要なのに
     --yes が無い場合を含む)
```

## 8. テスト・配布方針

- **前提検証テスト**: 6.3の実機検証項目(SAVEPOINT非対応、abort挙動、ロック挙動、WAL、COPY TO残留等)をpytestの自動テストとして保守し、`duckdb` パッケージのバージョン更新で設計前提が変わったことを検知できるようにする。
- **ui.dbスキーマのフィクスチャテスト**: 実際のDuckDB UIで作成したui.dbをフィクスチャとして保持し、非公式スキーマの変更(6.2#1)をCIで検知する。
- **ゴールデンHTMLテスト**: 代表的なnotebook(正常セル・エラーセル・abortセル・大量行・NULL/ネスト型)に対する生成HTMLのスナップショット比較。
- **CIマトリクス**: サポートする複数の `duckdb` パッケージバージョンでテストを実行する。
- **配布**: PyPIパッケージ(pip / uv tool)として配布する。`duckdb` 依存には**上限を固定しない**(上限を固定すると、より新しいDuckDBで作られたui.db・対象DBのstorage versionを開けなくなるため)。storage version不整合で開けない場合は「ツール側のduckdbパッケージ更新」を促す明確なエラーメッセージを出す(6.2#10)。
- **ライセンス**: MIT(全依存が許諾的ライセンスであるため。詳細はLICENSEファイルを参照)。
- **対応プラットフォーム**: Phase 1は**Linux/macOSのみ正式サポート**とする(ADR-008)。Windowsはコード上排除しないが、ロック中ui.dbのコピー可否が未検証(6.2#8)のためREADMEに「未検証」と明記し、検証はPhase 1.5以降のタスクとする。CIマトリクスもLinux/macOSのみで構成する。
